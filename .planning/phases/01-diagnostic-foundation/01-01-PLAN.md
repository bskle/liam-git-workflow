**4. `## Trigger Conditions` 区块（来自设计文档 §6）：**
- 自动触发：Git 远程命令（push/pull/fetch/ls-remote）返回非零退出码
- 同时检查 stderr 中以下失败模式：host resolution errors、connection timeout/refusal、TLS/SSL failures、proxy failures、authentication failures（403/401）、repository not found（404）、permission denied、protected branch/policy rejection
- 手动触发：用户通过 `liam-git-workflow-remote-diagnose` 命令或直接路由调用

**5. `## Diagnostics Order` 区块（来自设计文档 §7，五层，必须标号）：**
- **第 1 层：Local Repository State** — 检查当前分支、detached HEAD、进行中的 merge/rebase、未解决的冲突、upstream 是否存在、工作区状态。目的：在分析网络之前先排除本地状态阻塞。
- **第 2 层：Remote Target Validation** — 检查已配置的 remote、remote URL、分支 upstream 映射、remote 命名、owner/repo 目标的基本合理性。目的：提前捕获错误的 remote 目标、缺失 upstream、格式错误的仓库引用。
- **第 3 层：Authentication and Authorization** — 检查传输类型（HTTPS 或 SSH）、凭据助手配置、可识别的认证错误签名、从 CLI 输出可见的仓库/分支访问错误。目的：将身份和权限问题与传输/网络问题分离。
- **第 4 层：Network Path** — 检查 Git 代理配置、shell 环境代理变量、DNS 解析、HTTPS 路径的 TCP 443 可达性、最小 HTTPS 访问探测、将 `git ls-remote` 作为窄带远程访问测试。目的：识别命令行网络路径问题（代理不匹配、DNS 故障、超时、TLS 问题、公司过滤）。
- **第 5 层：Repository Policy and Server-Side Rules** — 检查受保护分支拒绝、禁止直接推送限制、仓库重命名或移除、组织策略消息。目的：避免将服务端策略误分类为客户端网络故障。

**6. `## Output Contract` 区块（来自设计文档 §8，包含所有必填字段）：**
诊断能力必须返回结构化结果，包含以下字段：
- `problem_category` — 问题类别（从故障分类中选择）
- `evidence` — 观察到的信号和 CLI 输出摘录
- `likely_cause` — 可能原因（基于信号→原因映射）
- `confidence` — 置信度（high / medium / low）
- `actions_taken` — 已执行的诊断步骤
- `recommended_next_action` — 建议的下一步修复操作
- `human_interaction_required` — 是否需要人工交互（boolean）
- `human_action_detail` — 如需人工交互，用户需执行的精确步骤及需回报的结果

**7. `## Human Interaction Rules` 区块（来自设计文档 §10）：**
- 仅在以下信息不可从 CLI 环境观测时询问用户：浏览器会话/仓库页面访问状态、GitHub 组织授权页面、VPN 状态、代理客户端状态、用户对轮换/替换凭据的批准、仓库外部的系统证书/公司设备策略。
- 向用户提问时，提示必须包含：(1) 为什么此步骤无法自动验证，(2) 用户应检查/执行的精确操作，(3) 用户应回报的精确信息。

**8. `## Response Style` 区块：**
- 陈述检测到的故障类别及置信度
- 按五层顺序列出已执行的检查及发现
- 清晰地给出建议的下一步操作（主 Agent 可直接执行）
- 如需要，明确说明人工交互需求及原因

用一个 Markdown 表格列出所有远程操作故障类别。每个条目包含类别名称、典型信号特征和受影响的操作（push/pull/fetch/ls-remote）。必须包含以下几个类别（来自设计文档 §7 各层）：

| 类别 | 信号特征 | 受影响操作 | 诊断层 |
|------|---------|-----------|--------|
| 本地状态阻塞 | detached HEAD, merge/rebase 进行中, 未解决的冲突, 无 upstream | push, pull | 第 1 层 |
| 远程目标错误 | remote URL 无效, upstream 不存在, 仓库名/owner 错误 | 所有 | 第 2 层 |
| 认证失败 | HTTP 401/403, SSH permission denied, 凭据助手为空或过期 | 所有 | 第 3 层 |
| 授权不足 | 仓库访问被拒, 组织策略限制, 分支保护 | push, fetch | 第 3 层 |
| DNS 解析失败 | host not found, 无法解析 Git 服务域名 | 所有 | 第 4 层 |
| 连接超时/拒绝 | connection timed out, connection refused | 所有 | 第 4 层 |
| TLS/SSL 错误 | certificate verify failed, SSL handshake error | 所有 | 第 4 层 |
| 代理错误 | proxy CONNECT 失败, 代理认证要求 | 所有 | 第 4 层 |
| 仓库策略拒绝 | protected branch, direct push forbidden, 仓库已归档/只读 | push | 第 5 层 |
| 仓库不存在 | repository not found (404), 仓库已重命名或删除 | 所有 | 第 5 层 |

**章节 2：** `## 五层诊断清单`

对每一层（来自设计文档 §7），以清单格式列出具体检查命令和预期输出：

- **第 1 层：本地仓库状态** — 检查命令列表（`git status`, `git branch`, `git rev-parse --abbrev-ref HEAD`, 检查 `.git/MERGE_HEAD` 等），每项检查的预期结果，非预期结果的解释。
- **第 2 层：远程目标验证** — 检查命令列表（`git remote -v`, `git branch -vv`, `git ls-remote --heads <remote>`），URL 格式验证规则。
- **第 3 层：认证与授权** — 检查命令列表（`git config --get remote.origin.url` 以确定传输类型, `git config --get credential.helper`, 常见认证错误消息及其含义）。
- **第 4 层：网络路径** — 检查命令列表（`git config --get http.proxy`, `env | grep -i proxy`, `nslookup github.com`, `Test-NetConnection github.com -Port 443`（Windows）/ `curl -I https://github.com`），每个命令的成功/失败含义。
- **第 5 层：仓库策略与服务端规则** — 常见服务端错误消息模式及其含义（受保护分支、推送限制、仓库归档），非 CLI 可检查项（浏览器验证 GitHub 设置）。

**章节 3：** `## 信号到原因映射`

一个映射表，将特定错误信号映射到可能的原因（按可能性排序）。格式：

```markdown
| 信号 | 可能原因 | 置信度 | 诊断层 | 验证方法 |
|------|---------|--------|--------|---------|
| `fatal: unable to access 'https://...': Could not resolve host` | DNS 解析失败 | high | 第 4 层 | `nslookup` 目标主机 |
| `fatal: Authentication failed for 'https://...'` | 凭据过期或错误 | high | 第 3 层 | 检查凭据助手 |
| `error: failed to push some refs to 'https://...'` + "protected branch" | 分支保护规则 | high | 第 5 层 | 浏览器检查仓库设置 |
| `ssh: connect to host github.com port 22: Connection timed out` | SSH 端口被防火墙阻止 | high | 第 4 层 | 测试 TCP 443（HTTPS）替代方案 |
| `fatal: unable to access '...': schannel: next InitializeSecurityContext failed` | Windows TLS/Schannel 问题 | medium | 第 4 层 | 检查系统证书、代理 |
```

至少包含 8 种不同的错误信号和映射。

**章节 4：** `## 标准修复动作`

按诊断层分类，每组列出具体的可执行修复命令：

- **第 1 层修复：** 完成/中止 merge/rebase（`git merge --abort`, `git rebase --abort`），设置 upstream（`git branch --set-upstream-to=origin/<branch>`），提交或暂存变更。
- **第 2 层修复：** 修正 remote URL（`git remote set-url origin <correct-url>`），添加 missing remote。
- **第 3 层修复：** 重新认证（触发凭据提示），切换传输类型（HTTPS ↔ SSH），清除过期凭据缓存。
- **第 4 层修复：** 配置/清除代理（`git config --global http.proxy`, `unset http_proxy`），切换端口（SSH 22 → HTTPS 443），使用 `ssh -T git@github.com` 测试 SSH 连通性。
- **第 5 层修复：** 用户需在浏览器中检查仓库设置、请求访问权限、检查组织策略——此层大部分为人工修复，CLI 手段有限。

**章节 5：** `## 升级与人工交互边界`

定义升级边界（来自设计文档 §10）：

- 列出 CLI 不可观测的信息及无法自动验证的原因：
  - 浏览器登录会话状态 — 无 CLI 可观测信号
  - GitHub 组织授权页面 — 需浏览器交互
  - VPN/代理客户端图形界面状态 — 无标准 CLI 接口
  - 操作系统/公司设备策略（系统证书、组策略） — 超出仓库 CLI 范围
  - 用户对凭据轮换/替换的批准 — 需用户确认
- 对于每种情况，提供精确的用户提示模板（包含三个部分：为什么无法自动化、精确的操作步骤、回报的精确信息）。
